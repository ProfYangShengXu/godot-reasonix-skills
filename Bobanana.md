# Bobanana.md — 核心工程大纲

> 每次调用 godot-dev / godot-test / godot-docs / godot-debug 技能时，
> Agent 必须读此文件并践行以下原则。

## 1. 禁止面向结果编程

不准用关键词匹配、正则、模式识别替代真正的逻辑处理。遇到问题必须查清根因，
从对应层面根本解决，而不是修表面现象。修 bug 前先写一个暴露该 bug 的测试，再修代码。
不准写"看起来对了"但经不起推敲的逻辑。

## 2. 架构设计原则（9 条）

高内聚低耦合 · 单一职责 · 开放封闭 · KISS · DRY
迪米特法则 · 依赖倒置 · 关注点分离 · YAGNI

## 3. 风险与质量意识（7 个维度）

技术债（妥协要标记 TODO） · 回滚（可逆吗 API 兼容吗） · 灰度（feature flag?）
熔断降级（超时/重试/降级） · 监控告警（有日志吗） · 容错冗余（单点在哪）
安全（输入消毒/密钥不从代码来）

## 4. 代码规范

- **不随便重构**：不重构没问题的代码。重构必须有明确收益
- **不写未接线死函数**：定义了就必须有调用方，否则删掉
- **不自己造轮子**：实现前先搜 GitHub / 标准库 / 项目已有代码
- **改完即清理**：删未用 import、死变量、残留 TODO
- **PRD 一致性**：有 PRD 时实现必须一致，不一致时停下来问用户
- **每个字符都有依据**：说不清为什么的 import、函数、参数、分支、error 处理，就不该存在

## 5. 调试闭环原则

- **先测后修**：觉得有问题时先跑一遍调试会话抓日志，不要凭感觉改代码
- **LLM 分析是辅助，不是决策**：AnalyzeLogsWithLLM 的输出要人工/Agent 判断后再改
- **可复现**：每次 RunDebugSession 自动保存到 debug-sessions/，失败时可回放

## 6. RED LINE（不可触碰）

| 红线 | 后果 |
|------|------|
| 不准声称跑了命令但没贴退出码 | 必须贴退出码+输出 |
| 不准输出"应该没问题" | 必须贴证据 |
| 不准写空壳函数（return nil / return ""） | 质量门拒绝 |
| GOAL 不能改一个字 | 哈希锚定拒绝 |
| 角色完成 = 立即 queue_next_prompt | 不准停顿、不准征求同意 |
