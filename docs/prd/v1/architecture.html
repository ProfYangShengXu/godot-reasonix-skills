1. 架构总览

核心思路

技能集不是一个运行时服务，而是一组 Reasonix Skill Tool 定义 + 一套 文件操作库。 Agent 在 Reasonix 环境中加载本技能集后，获得 7 个模块约 15 个工具接口。 用户说自然语言 → Agent 规划 → 调用工具 → 文件写入 + Godot CLI 验证 → 项目就绪。

关键架构决策

决策选择理由
交互方式文件系统 + CLI Godot 无 GUI 远程控制协议。文件操作是唯一跨平台可靠的途径
技能交付形态 Reasonix Skill (Playbook) 与产品定位一致：Agent 调用技能集。不需额外服务进程
脚本语言 GDScript (非 C#) 产品 PRD 面向 Godot 新手，GDScript 学习成本最低；C# 做可选扩展
是否维护编辑器插件 不做 产品 PRD 明确"不做可视化编辑器替代品"，编辑器插件属于 GUI 范畴
模板分发 内嵌在技能集中（复制而非引用） 每个 fork 是独立副本，用户可任意修改，不受上游模板变更影响

2. 模块分解

按能力边界拆为 7 个模块。选择理由：每个模块的职责可以用一句话说清， 可以独立实现和测试。M1-M4 是原子能力，M5 是组合产物， M6 是基础设施，M7 是文档。

M1 · Project Scaffolder P0

一句话职责：创建一个能被 Godot 4.x 打开的合法项目目录。

创建 project.godot（含项目名、渲染器、分辨率）、目录骨架（scenes/ scripts/ assets/…）、默认空场景、占位 icon。是其他所有模块的前置条件。

为什么独立成模块：项目初始化是明确的一次性操作，与场景/脚本无关。单独测试（验证 project.godot 语法）。

M2 · Scene Generator P0

一句话职责：将结构化节点描述转换为合法的 .tscn 文件。

创建含任意 2D 节点树的场景、向已有场景增删节点、修改节点属性。核心能力是维护 .tscn 文本格式的合法性（缩进、uid、资源引用）。

为什么独立成模块：.tscn 格式有独立语法规则，与 GDScript 完全不同。需要一个专门的渲染器维护 TSON 格式。

M3 · Script Manager P0

一句话职责：创建 .gd 文件并挂载到场景节点上。

生成函数体和信号定义、将脚本附加到场景节点的 script 属性、建立信号连接。与 M2 紧密协作但职责分离：M2 管"场景结构"，M3 管"行为逻辑"。

为什么独立成模块：GDScript 的语法验证、代码生成、挂载机制是独立领域。单一职责要求它不与场景逻辑混在一起。

M4 · Resource Resolver P0

一句话职责：确保外部资源文件（图片/音频/字体）被 Godot 正确索引。

将用户提供的资源文件复制到项目 assets 目录、返回正确的 res:// 路径、维护资源路径与文件系统路径的双向映射。

为什么独立成模块：资源管理和场景/脚本是正交问题。不依赖 M2/M3，可以在已有项目上独立使用。

M5 · Template Library P0

一句话职责：提供三款可运行的 2D 游戏模板供用户 fork。

预置「平台跳跃」「打砖块」「点击收集」三个完整项目。用户 ForkTemplate → 获得可运行的独立游戏副本，之后可自由修改。

为什么独立成模块：模板是 M1-M4 + M6 的组合产物。它依赖所有子模块，但提供的是更高层的「复制项目」抽象。

M6 · Godot CLI Bridge P1

一句话职责：封装 godot 可执行文件的所有 CLI 操作。

统一管理 godot 进程生命周期：参数构造、路径解析、超时/kill、退出码解析、版本检测。M5 用它验证模板，Agent 用它运行/导出项目。

为什么独立成模块：进程管理（超时、SIGKILL、跨平台路径）是独立的技术关注点，不宜散落在其他模块中。

M7 · Skill Playbook & Documentation P1

一句话职责：让 Agent 知道怎么用 M1-M6。

结构化 API 说明 + 自然语言→工具链的 Few-shot 示例 + 能力边界声明的 Markdown / YAML 文档。

为什么独立成模块：文档独立于实现，可单独审查、单独版本管理。

3. 依赖关系

M1 (Scaffolder) — 无依赖，独立 M6 (CLI Bridge) — 无依赖，独立 │ M2 (Scene Generator) ──→ M1 （场景需要项目骨架） M3 (Script Manager) ──→ M2 （脚本挂载需要场景） M4 (Resource Resolver) ──→ M1 （资源导入需要项目路径） │ M5 (Template Library) ──→ M1 + M2 + M3 + M4 + M6 M7 (Documentation) — 无代码依赖

依赖规则满足性：

  • ✅ 高层 → 接口层 ← 低层实现：M5 调用 M1-M4 的接口，不调用内部实现
  • ✅ 不允许低层 → 高层：M2 不会调用 M5，M3 不会调用 M5
  • ✅ 无循环依赖：M1→M2→M3 是线性链，M4 只依赖 M1，M5 是汇集点

4. 接口定义

完整接口签名见 docs/prd/v1/prd.yaml 各模块的 interfaces 段。以下是概览：

模块接口输入要点输出
M1 ScaffoldProject root_path, project_name, resolution, renderer project_root | error
M2 CreateScene / AddNode / RemoveNode / SetNodeProperty scene_path, node 结构（type, name, position, children） scene_path | error
M3 CreateScript / AttachScript / ConnectSignal script_path, extends, functions[], scene_path, node_path script_path | error
M4 ImportResource / ResolvePath source_path, dest_type, project_root resource_path | error
M5 ListTemplates / ForkTemplate / CustomizeTemplate template_id, target_path, overrides{} project_root | error
M6 RunGodot / CheckGodotVersion project_root, args[], timeout_sec stdout, stderr, exit_code | error
M7 GetSkillDoc / FindPlaybook / GetScopeBoundary intent（查询语义） playbook | scope

5. 风险评估

维度风险描述涉及模块
技术债.tscn 格式无官方 schema，每次 Godot 小版本可能变更节点属性名M2
技术债GDScript 语法在 4.0→4.1→4.2 间有破坏性变更（@export 语法）M3
技术债三个模板需随 Godot 同步更新M5
回滚每次操作生成文件 diff，Agent 通过 git revert 或文件快照实现撤销全局
灰度技能集无灰度概念，但模板可标记"实验性"标签M5
熔断Godot CLI 调用须设超时，超时后 SIGTERM → 1s → SIGKILLM6
监控所有 Godot CLI 调用的 stderr/stdout 应捕获并存档用于调试M6
容错目录已存在合法 Godot 项目 → 复用而非覆盖M1
安全路径参数 sanitize（防 ../ 穿越）M1, M6
安全用户提供的 GDScript 代码片段不做直接 eval，用模板拼接M3

无风险模块说明：M7（文档）不涉及运行时，无以上七维风险。

T5 · 中 M6 Godot CLI 桥

7. 实现指引

推荐实现顺序

  T1 (M1 脚手架) ──────────────── 基础，无依赖
  T5 (M6 CLI Bridge) ─────────── 基础，无依赖，尽早可验证 godot 可用
       │
  T2 (M2 场景生成器) ─── 核心难点，先攻
  T4 (M4 资源解析器) ─── 简单，与 T2 可并行
       │
  T3 (M3 脚本管理) ───── 依赖 T2
       │
  T6 (M5 模板) ──────── 依赖 T1-T5，验证整体
  T8 (S 层测试) ─────── 依赖 T6
  T7 (M7 文档) ──────── 最后，但可提前写框架

技术选型建议（FREE ZONE）

  • TypeScript（与 Reasonix 运行时的语言一致）
  • .tscn 生成用模板字符串 + 逐行构建器，而非 XML DOM（.tscn 不是 XML，是自定义 TSON）
  • GDScript 生成用模板引擎（如 EJS 风格），避免 AST 操作增加复杂度（YAGNI）
  • 测试框架：vitest（TypeScript 生态）
  • Godot 版本锁定：GitHub Action 中用 4.3-stable Docker 镜像做 CI
