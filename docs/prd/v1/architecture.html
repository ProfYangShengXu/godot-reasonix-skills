1. 架构总览

核心理念

在原有技能集（M1-M7）的基础上，新增 4 个模块 将技能集从 「单向编码管道」升级为 「写 → 跑 → 测 → 修」的调试闭环。

新模块不改变原有 M1-M7 的任何接口。它们作为平行的新能力层， 通过 M6（Godot CLI Bridge）与原有模块对接，保持架构整洁。

新增架构决策

决策选择理由
日志解析方式正则 + 行模式 Godot 日志格式半结构化，AST 太重量级（YAGNI），逐行正则轻量可增量
输入模拟方式操作系统 API Godot 无 --headless 输入模拟支持。平台 API 不修改游戏项目，通用性最强
LLM 分析策略可选 + 可降级 LLM API 有成本/延迟/失败风险。默认开启但可关闭，失败时 fallback 到结构摘要
会话存储项目本地 debug-sessions/ 与项目绑定、自然隔离、已 .gitignore 不泄露

2. 新增模块分解

M8 · Log Capture & Parser P0 NEW

一句话职责：捕获 godot 运行时的 stdout/stderr，解析为结构化日志条目。

通过 M6 启动 godot --headless 并捕获进程输出。解析引擎识别 Godot 错误格式、 GDScript print() 输出、traceback 行号，输出结构化的日志条目数组。 提供摘要统计和 LLM 友好的聚合报告。

接口： CaptureLogs · ParseLogText · SummarizeLogs

为什么独立成模块：日志捕获和解析是调试流水线的数据入口。与输入模拟（M9）、编排（M10）职责分离，可独立测试和复用。比如用户可单独用 ParseLogText 解析已有的日志文件。

M9 · Input Simulator P0 NEW

一句话职责：向运行中的 Godot 游戏窗口模拟键盘输入。

通过操作系统级 API（Windows SendKeys、Linux xdotool、macOS osascript）发送键盘事件。 支持三种原语：press（按下）、release（释放）、tap（按下+释放）。 支持将按键序列保存为可复用的输入脚本（JSON），支持回放。

接口： SimulateKeySequence · CreateInputScript · ReplayInputScript

为什么独立成模块：跨平台输入模拟是一个独立的技术关注点（三套 API、窗口焦点管理、延迟/重试），不宜与日志处理混在一起。

M10 · Debug Orchestrator P0 NEW

一句话职责：编排完整调试会话——启动游戏、模拟输入、捕获日志、LLM 分析。

这是 Agent 调试的入口点。将 M6+M8+M9 组合为一条命令： 「启动游戏 → 等待 N 秒 → 输入模拟操作 → 关闭游戏 → 收集日志 → LLM 分析 → 出报告」。 LLM 分析将日志 + 关联的 .gd 代码片段打包为 Prompt，输出根因定位和修复建议。

接口： RunDebugSession · AnalyzeLogsWithLLM · ListDebugSessions

为什么独立成模块：编排是组合逻辑，与原子能力（M8/M9）分离。如果以后有新的原子能力（如截图），M10 可扩展而不修改底层。

M11 · GDScript Log Utilities P1 NEW

一句话职责：提供 log_info/log_warn/log_error 的 GDScript 模板。

生成可直接复制到 Godot 项目的 log_utils.gd 模板脚本。 统一输出格式 "LOG:[LEVEL]:[message]" 方便 M8 精准解析。 还提供 InjectLogStatements 工具，自动在现有 .gd 文件的关键函数中插入日志。

接口： GenerateLogScript · InjectLogStatements

为什么独立成模块：GDScript 模板生成是内容创作，不是运行时逻辑。与 M8（解析）严格对称——M11 负责「写日志」，M8 负责「读日志」。

3. 依赖关系

原有模块: M1 (Scaffolder) — 无依赖，独立 M6 (CLI Bridge) — 无依赖，独立 M2 (Scene Generator) ──→ M1 M3 (Script Manager) ──→ M2 M4 (Resource Resolver) ──→ M1 M5 (Template Library) ──→ M1+M2+M3+M4+M6 M7 (Documentation) — 无代码依赖 新增模块: M8 (Log Capture) ──→ M6 (通过 M6 运行 godot) M9 (Input Simulator) ──→ M6 (启动 godot 进程) M10 (Debug Orchestrator) ──→ M6 + M8 + M9 (组合编排) M11 (GDScript Log Utils) — 无代码依赖 (纯模板生成)

依赖规则满足性：

  • ✅ M8-M10 只依赖 M6（基础设施层），不依赖 M1-M5（业务模块）—— 可独立开发测试
  • ✅ M10 是组合层，不直接操作文件系统，通过 M8/M9 间接操作 —— 单一职责
  • ✅ 无循环依赖 —— M8→M6, M9→M6, M10→M8+M9+M6 是线性向上
  • ✅ M11 无任何代码依赖，可首批实现

5. 风险评估（新增模块）

维度风险描述涉及模块
技术债Godot 日志格式在版本间可能有差异，解析器正则需维护M8
技术债跨平台输入模拟（Win/Linux/macOS）三套实现需分别维护M9
回滚RunDebugSession 每次执行应保存完整 session 记录，便于追溯M10
灰度LLM 分析属于"增强功能"，可以按用户配置开启/关闭M10
熔断LLM API 调用超时（常见 30s+），需设超时 + fallbackM10
熔断输入模拟可能因窗口未聚焦而失败，需重试 + 超时退出M9
监控日志风暴（游戏每帧输出）可能撑爆内存，需行数上限保护M8
容错Godot 进程可能提前退出（崩溃），需优雅处理 partial 日志M8, M10
安全输入模拟不能作用于非 Godot 窗口，需窗口标题校验M9
安全日志可能包含敏感信息（API key 被 print），LLM 分析前需脱敏M10

无风险模块说明：M11（GDScript 日志模板）是纯文本模板生成，不涉及运行时操作，无以上七维风险。

7. 实现指引

推荐实现顺序

  T9  (M11 日志模板)  ─── 无依赖，先做，立即能给模板项目用
  T10 (M8 日志解析器) ─── 纯文本解析，无依赖，可独立开发测试
       │
  T11 (M8 日志捕获) ──── 依赖 T10 + M6(已有)
  T12 (M9 输入模拟器) ── 依赖 M6(已有)，与 T11 并行
       │
  T13 (M10 编排器) ──── 依赖 T11 + T12，调试入口
       │
  T14 (模板日志埋点) ─── 依赖 T9 + T12，让模板可调试
  T15 (S 层端到端) ──── 依赖 T13 + T14，验收闭环

技术选型建议（FREE ZONE）

  • 日志解析：用 RE2 兼容的正则（TypeScript 原生 RegExp），Godot 错误格式用多级匹配器链（先匹配 ERROR 行 → 再匹配后续 traceback 行）
  • 输入模拟：Windows 用 child_process.spawn('powershell', ['-Command', ...]) 调用 SendKeys；Linux 用 xdotool；macOS 用 osascript -e 'tell application "System Events"...'
  • LLM 分析：用 fetch API 直接调用 OpenAI/Anthropic REST API，不引入额外 SDK 依赖（避免依赖膨胀）
  • 会话存储：JSON 格式存储在 debug-sessions/<timestamp>-<session_id>.json
  • 测试：M8 的解析器用静态日志文件做单元测试（Mock 数据），不依赖 godot 进程。M9 用 Mock 层屏蔽平台差异

Log Entry 数据结构（M8 核心类型）

  interface LogEntry {
    level: "error" | "warning" | "info" | "debug";
    message: string;           // 日志消息正文
    file?: string;             // res://scripts/player.gd
    line?: number;             // 42
    timestamp: number;         // 相对运行起点毫秒
    raw: string;               // 原始日志行
    category?: "script" | "engine" | "print" | "system";
  }

  interface LogSummary {
    stats: { total: number; errors: number; warnings: number; infos: number; };
    top_errors: Array<{
      file: string;
      line: number;
      count: number;
      example: string;
    }>;
    timeline: Array<{ ms: number; event: string }>;
    diagnosis_hints: string[];
  }

Input Action 数据结构（M9 核心类型）

  interface InputAction {
    action: "press" | "release" | "tap";
    key: string;             // "Space" | "W" | "A" | "S" | "D" | "Up" | ...
    delay_ms: number;        // 执行此动作前的等待时间（相对上一动作）
  }

  interface InputScript {
    name: string;
    created: string;         // ISO timestamp
    sequence: InputAction[];
  }
