🎮 Godot Reasonix 技能集

让编程 Agent 操控 Godot 引擎，零门槛开发游戏
产品设计文档 v1.0 · 非技术方案 · 后续由 pipeline 架构师转换为技术 PRD

1. 背景与目标

现实问题

大量有编程能力但没有游戏开发经验的人（前端/后端工程师、学生、独立创作者）想用 Godot 做游戏， 但 Godot 编辑器本身是一个复杂的 IDE 类图形界面——场景树、节点系统、信号连接、GDScript 调试…… 学习曲线陡峭。即使会写代码，也要花数周熟悉编辑器操作和工作流。

另一方面，编程 Agent（Cursor / Windsurf / Reasonix 等）已经能写代码、操控命令行、操作文件系统， 但它无法「看见」Godot 编辑器的画布、场景树、资源面板——它能生成 .tscn 和 .gd 文件， 但缺少一个技能层把「自然语言需求」转换成「Godot 项目可执行的操作序列」。

一句话定位

「像 VSCode Copilot 之于代码 —— 但之于 Godot 编辑器。」

更准确地说：一套 Agent 可调用的技能集 + 脚手架，让用户用自然语言描述游戏想法，Agent 自动完成 Godot 项目的搭建、场景编排、脚本编写、资源管理。

成功标准（非技术指标）

  • 新手首款游戏用时 < 2 小时：一个完全没碰过 Godot 的人，从安装到跑起第一个可玩的 2D 小游戏
  • 技能集采纳率 > 30%：在目标用户群（Reasonix / Cursor 用户）中，下载后一周内持续使用
  • 用户自助率 > 70%：用户提问中，Agent 在不求助人类的情况下独立完成操作的比例
  • NPS > 40：首批用户净推荐值

👤 画像 A：独立游戏作者 · Solo Dev

维度说明
身份/场景全职或业余独立开发者，会写代码（Python/JS/Go），想快速验证游戏创意
痛点Godot 编辑器不熟悉，拖拽节点、调资源、连信号远不如写代码顺手；把时间花在学编辑器而非做游戏
使用频率每日 / 每周密集使用
核心诉求「我描述游戏逻辑，帮我搭好场景、写好脚本、配好资源路径，我只需要调参数和改文案」

👤 画像 B：编程 Agent 使用者 · AI Native Dev

维度说明

痛点Agent 能写 .gd 脚本，但无法操控编辑器——创建节点、设置碰撞层、导入音效这些事 Agent 做不到，人得自己进编辑器做
使用频率每次开发会话（每天多次）
核心诉求「Agent 能完全操控 Godot 项目——创建/删除/修改场景、管理资源、运行调试、导出打包。我只需要在对话里说『加个跳跃功能』」

👤 画像 C：游戏设计学习者 · Student

维度说明
身份/场景在校学生或转行学习者，编程基础薄弱但想学游戏开发
痛点官方文档和教程以编辑器操作为主，学起来慢；想先看到成果再理解原理
使用频率每周数次
核心诉求「用大白话描述一个游戏，Agent 帮我做出来，我边玩边看代码学」

3. MVP 定义

🥇 Must-have（MVP 必须包含）

  • P0 项目脚手架生成 —— 一键创建 Godot 4.x 项目目录结构、project.godot 配置、默认场景
  • P0 场景文件（.tscn）生成与编辑 —— 用文本方式创建/修改 2D 场景（节点树、Transform、Sprite、CollisionShape）
  • P0 GDScript 脚本生成与挂载 —— 写 .gd 文件、自动挂载到场景节点、建立基础信号连接
  • P0 资源路径管理 —— 自动处理纹理/音效/字体的导入路径，确保 Godot 资源面板可识别
  • P0 基础 2D 游戏模板 —— 内置「平台跳跃」「打砖块」「点击收集」三个模板，用户可 Fork 修改
  • P1 Godot 命令行调用 —— godot --headless 导出、运行测试、语法检查
  • P1 技能集文档 + 示例 Playbook —— Agent 可读的 API 说明 + 自然语言→操作步骤的映射示例

📅 Nice-to-have（一期后加）

  • P2 3D 场景支持（Spatial/Node3D、Mesh、Light、Camera 节点）
  • P2 动画（AnimationPlayer + AnimationTree 的脚本化创建）
  • P2 TileMap 编辑器自动化（刷 TileSet、Layers、自动碰撞）
  • P2 资源管线（自动导入 Spritesheet 分割、音频转码 .ogg）
  • P3 多人游戏网络同步脚手架（RPC 配置、NetworkedSpawner）
  • P3 移动端导出 + 商店发布辅助（打包签名、图标生成）

🚫 明确不做什么

  • 不做 可视化编辑器替代品 —— 我们不重写 Godot Editor UI。Agent 通过文件操作 + 命令行与 Godot 交互，用户仍可随时打开编辑器手动微调
  • 不做 美术资产生成 —— 不生成图片/模型/音效。只管理资源引用路径
  • 不做 游戏逻辑引擎 —— 不替代 Godot 的 Physics / Animation / Audio 运行时。只做「搭架子」
  • 不做 低代码拖拽界面 —— 目标用户是「写代码的人 + AI Agent」，不需要图形化搭建
  • 不做 Godot 版本兼容 —— MVP 锁定 Godot 4.x（当前最新稳定版），不向下兼容 3.x

4. 2B / 2C 分析

维度 🎯 To C（个人开发者） 🏢 To B（工作室 / 教育机构）
用户是谁 独立开发者、AI 重度用户、游戏开发初学者 小型游戏工作室教学组、培训机构、Game Jam 组织方
付费意愿 中等——愿意为「节省时间」付费，但单价敏感（免费/低价更优） 高——如果能量化提升团队效率，愿意付年费
决策链 单人决策——自己觉得有用就下载 组长/讲师评估 → 试用 → 批量采购
核心功能差异 快速原型 + 模板 + Agent 自由操控 模板库 + 批量项目脚手架 + 培训演练沙盒 + 版本锁定（确保学生环境一致）
获客渠道 GitHub 开源 / X / Reddit / 独立开发者社区 直接 Outreach + 机构合作 + 案例白皮书

主次关系

To C 先行，To B 后置。

MVP 阶段完全面向个人开发者（To C）。技能集以开源/免费形式发布，积累社区口碑和模板生态。 当 C 端覆盖达到一定规模（GitHub Star > 1k、Discord 社区 > 500 人），自然会出现 B 端需求， 届时推出付费版（企业许可证 / 私有部署 / SLA 支持）。

功能分区：C 端核心是「自由 + 速度」——Agent 能做任何事；B 端额外需要「可控 + 可追溯」——操作日志、项目模板锁定、权限分角色。

5. 设计理念

体验关键词

Agent-Native 零摩擦 黑盒可选

  • Agent-Native（Agent 原生） —— 技能集的每个接口都是为 Agent 调用设计的：输入输出结构化、错误信息机器可读、操作幂等。人类可读的注释和文档是第二优先级。
  • 零摩擦（Zero Friction） —— 从「我有个想法」到「项目跑起来」不超过 3 条指令。不需要安装额外 GUI、不需要配置环境变量（除 Godot 路径外）。
  • 黑盒可选（Optional Transparency） —— 用户可以选择只看结果，也可以让 Agent 展示每一步创建的代码和场景文件。新手看结果，进阶用户审查过程。

核心交互原则

  1. 自然语言入，完整项目出。 用户的每条游戏描述，Agent 应当能转化为一组对 Godot 项目文件的精确写入，而非返回「我建议你打开编辑器做 X」。
  2. 可撤销与可预览。 每次操作前生成 diff，用户可批准/回滚。不盲目改动已有项目结构。
  3. 场景即代码，代码即场景。 .tscn 和 .gd 都是纯文本，Agent 视它们为同一层级的产物——不区分「搭场景」和「写逻辑」，统一用工具操作。

视觉风格方向

这是 Agent 技能集，不是用户界面产品。不需要 GUI 设计。
唯一面向用户的「界面」是：

  • 终端输出（操作日志）：灰底白字，绿色 ✓ 成功 / 红色 ✗ 失败 高亮
  • 项目文件变化：标准的 git diff 风格展示
  • 模板代码：遵循 Godot 官方 GDScript 风格指南（snake_case 函数/变量、PascalCase 节点名）

整体调性：专业、克制、可预测——让用户信任 Agent 生成的项目质量。

6. 风险（非技术）

📉 市场风险

竞品分析： 目前没有直接竞品。间接竞品包括：

  • Godot 官方教程 / 文档 —— 免费但靠人学，不是 Agent
  • GameMaker / RPG Maker —— 低代码但封闭生态，Agent 无法接入
  • AI 写代码（Claude 直接写 .gd）—— 缺乏场景编排和资源管理，只能写逻辑

风险： 市场太小众（Agent + Godot 的交集）——如果 Godot 本身用户基数不够大，技能集的使用场景会受限。
缓解： Godot 4.x 发布后用户快速增长（Steam 调查占比已到 ~2-3%），且 AI 编程助手本身在高速扩张，两个曲线交叉点正在到来。

⚙️ 运营风险

  • Godot 版本碎片化： 4.0 → 4.1 → 4.2 → 4.3 节点/API 有破坏性变更。技能集需要紧跟每个小版本，否则生成的 .tscn 在新版打不开。
  • 合规 / 开源许可： Godot 本身是 MIT 许可。技能集生成的游戏代码归用户所有。但技能集自身如果用了 GDScript 代码片段做模板，需确保不引入传染性许可证。
  • Agent 行为不可预测性： 用户可能用自然语言让 Agent 执行超出技能集设计范围的操作（如「生成一个 3D MMORPG」），导致 Agent 输出质量下降、用户失望。需要清晰的 Scope 文档和 graceful fallback 提示。

📈 增长风险

冷启动： 技能集在无社区、无模板、无案例的情况下发布，用户下载后不知道能做什么。

  • 解法： 发布时自带 3 个完整游戏模板（平台跳跃、打砖块、点击收集），用户改改就能跑
  • 解法： 提供一段「从零开始的 5 分钟 Demo」视频/文字记录——用户看着 Agent 一步步生成一个完整小游戏
  • 解法： 在 Godot 社区（r/godot、Godot Discord）和 AI 编程社区（r/cursor、r/copilot）同步发布

💰 变现路径

MVP 阶段：免费 + 开源

  • 积累用户 → 积累模板库 → 积累品牌信任
  • 通过 GitHub Sponsors / 爱发电 接受捐赠

后续变现（用户数达标后考虑）：

  • 企业许可证： 工作室/培训机构需要锁定版本、操作日志、批量项目分发，付费解锁
  • 高级模板市集： 社区贡献的高质量游戏模板，平台抽成 15-20%
  • 云构建管道： GitHub Action + Godot 导出打包，一键生成 Windows/macOS/Linux/Web 构建（按构建次数计费）

核心逻辑： 先让价值被感知，再谈收费。变现不能影响核心体验。
