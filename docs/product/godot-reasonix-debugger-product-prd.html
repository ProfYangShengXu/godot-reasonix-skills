🔍 Godot Reasonix 游戏调试与测试模块

日志采集 + LLM 驱动的自动化游戏测试，让 Agent 真正「看见」游戏运行态
产品设计文档 v1.0 · 非技术方案 · 后续由 pipeline 架构师转换为技术 PRD

1. 背景与目标

现实问题

当前 Godot Reasonix 技能集可以创建项目、编写脚本、搭建场景、管理资源——但这是一个单向管道： Agent 写了代码，却无法知道代码跑起来对不对。

实际游戏开发中，超过一半的时间花在调试上：

  • 角色跳跃落地后卡在地板里 —— 是 CollisionShape 没配好，还是刚体参数不对？
  • 点击按钮没反应 —— 信号没连上？脚本没挂载？路径写错了？
  • 小球穿墙而过 —— Area2D 的 layer/mask 配置冲突？
  • 某个场景加载时崩溃 —— 资源路径失效？脚本编译报错？

这些问题，当前的 Agent 无法诊断。它「写完了」就无法再介入。
结果：用户还是要自己手动操作 Godot 编辑器去反复试错，Agent 只做了前半段工作。

一句话定位

「让 AI Agent 不仅能写 Godot 代码，还能『看见』游戏运行时、『操控』输入、『诊断』问题的闭环调试系统。」

更具体地说：通过日志采集模块捕获游戏运行态的结构化信息，通过游戏测试模块让 Agent 能像人一样操控游戏（按键盘、移动鼠标、截图比对），并用 LLM 自动分析日志 → 定位问题 → 修复代码 → 重新验证，形成完整的「写 → 跑 → 测 → 修」闭环。

成功标准（非技术指标）

  • 问题定位时间缩短 70%：Agent 辅助下，从发现问题到定位根因的时间相比纯手动调试减少 70%
  • 首次修复准确率 > 50%：Agent 自动诊断并修复的问题中，超过一半一次改对，无需回滚
  • 用户自助调试率 > 60%：用户遇到游戏运行时错误时，Agent 在不求助人类的情况下独立完成「捕获 → 诊断 → 修复」
  • 自动回归测试覆盖 > 80% 的核心玩法路径：对每个模板游戏的核心交互路径（跳跃、碰撞、收集），Agent 能自动录制并回放测试

3. MVP 定义

🥇 Must-have（MVP 必须包含）

  • P0 游戏运行日志捕获 —— 通过 godot --headless 运行游戏，完整捕获 stdout/stderr、GDScript print() 输出、Godot 引擎错误信息，输出为结构化文本
  • P0 日志解析与错误检测 —— 自动识别日志中的 Error/Warning 级别消息，关联到对应的 .gd 文件和行号，提炼出「什么类型的错误 × 哪个文件 × 哪一行 × 错误信息」的结构化报告
  • P0 基础键盘输入模拟 —— 通过 CLI 参数或暴露的 Godot API 向运行中的游戏发送按键事件（方向键、空格、Enter、WASD），让 Agent 能「操控」游戏角色
  • P0 日志 → 问题定位流水线 —— 将捕获的日志原文 + 结构化错误信息 + 相关脚本代码片段 + 项目结构上下文打包为 Prompt，调用 LLM 分析，输出问题定位和修复建议
  • P1 自动运行 + 日志收集 —— 一条命令完成「启动游戏 → 等待 N 秒 → 输入模拟操作序列 → 关闭游戏 → 收集全部日志」的完整流程
  • P1 GDScript 内嵌日志注解 —— 提供预设的日志模板/宏（如 log_info()、log_warn()、log_error()），让 GDScript 脚本产出格式统一的日志，便于后续结构化解析

📅 Nice-to-have（一期后加）

  • P2 鼠标点击/移动模拟 —— 在游戏中模拟鼠标事件（移动到坐标、点击、拖拽）
  • P2 游戏截图捕获 —— 运行时截取游戏画面，供 LLM 或用户查看
  • P2 LLM 视觉分析 —— 将截图 + 日志一起发给 LLM（如 GPT-4V），让模型「看」到游戏画面状态辅助分析
  • P2 自动修复建议 → 自动修复 —— Agent 分析出问题后，自动修改对应 .gd / .tscn 文件，再跑一轮验证
  • P3 回归测试套件录制与回放 —— 用户演示一次操作流程，Agent 录制为可重复执行的测试脚本
  • P3 游戏状态快照对比 —— 在关键帧保存游戏状态（变量值、Transform），回放时自动比对 Diff
  • P3 性能分析集成 —— 捕获 FPS、帧时间、内存使用曲线，识别性能瓶颈

🚫 明确不做什么

  • 不做 Godot 内置调试器的 GUI 替代 —— 我们不重写断点/步进/变量监视等功能。只做「无头模式下 Agent 可读」的日志和输入模拟
  • 不做 游戏内可视化 Debug Overlay —— 不向游戏画面添加 FPS/碰撞框/NavMesh 可视化（这些由 Godot 编辑器自带 Debug 选项管理）
  • 不做 第三方测试框架的封装 —— 不封装 GUT/WAT/gdUnit。模块输出原始日志 + 结构化数据，不定义新的断言 DSL
  • 不做 通用 GUI 测试工具 —— 不试图做成一个通用的 GUI 自动化测试平台（如 Selenium for Godot）。只服务于 Agent 调试游戏
  • 不做 移动端/主机端部署测试 —— MVP 只做桌面端（Windows/macOS/Linux）的无头测试

主次关系

To C 先行，To B 验证后再投入。

MVP 完全面向个人开发者。调试是每个游戏开发者的日常痛点，C 端需求真实且紧迫。 当 C 端用户达到一定规模后，工作室和教育机构会自然提出企业级需求（批量运行、CI 集成、报告聚合），届时再投入 B 端功能。

功能分区：C 端核心是「即时反馈」——跑一次看结论；B 端额外需要「规模化与可追溯」——批量跑、历史对比、角色权限。

5. 设计理念

体验关键词

闭环 可观测 LLM-Native

  • 闭环（Closed Loop） —— Agent 不再「写完就交差」。写代码 → 运行 → 捕获日志 → 分析 → 修复 → 再验证，形成一个完整的自动循环。用户只需要描述问题，Agent 负责跑完全程。
  • 可观测（Observable by Default） —— 每次运行默认产出完整的日志 + 结构化错误报告 + 输入操作序列记录。用户和 Agent 都能回溯「发生了什么」。不做「静默失败」——即使游戏没崩溃，也能看到所有运行时输出。
  • LLM-Native（为 LLM 设计） —— 所有输出格式（日志、错误报告、运行摘要）优先为 LLM 解析优化：结构化、扁平化、无冗余、携带文件路径和行号引用。人类可读性是第二优先级，但不应损害机器可读性。

核心交互原则

  1. 一次命令跑完全流程。 用户的调试请求应当对应一条命令：「运行并按 W 键 3 秒，看角色是否碰撞到墙壁」。不需要用户手动拼接启动 → 模拟 → 收日志多个步骤。
  2. 错误信息要「可执行」。 日志解析结果不能只是显示一堆文字——必须包含可操作的信息：「scripts/player.gd:42 行的 jump_velocity 引用了未定义的变量 gravity」。LLM 可以直接据此修改代码。
  3. 人机同权。 Agent 能做的事，用户用命令行也能做。不隐藏底层操作的 CLI 入口，进阶用户可精细控制参数。
┌──────────────────────────────────────────────────────┐ │ Agent 调试闭环流程 │ ├──────────────────────────────────────────────────────┤ │ │ │ ① 用户描述问题 │ │ ↓ │ │ ② Agent 解析 → 生成测试脚本（键盘操作序列 + 预期结果） │ │ ↓ │ │ ③ godot --headless 运行游戏 + 输入模拟 │ │ ↓ │ │ ④ 捕获日志 + 截图 + 错误信息 │ │ ↓ │ │ ⑤ LLM 分析日志：定位根因 + 关联代码文件 │ │ ↓ │ │ ⑥ Agent 修改 .gd / .tscn 文件 │ │ ↓ │ │ ⑦ 重新运行测试 → 确认修复 ✓ / 仍有问题 → 回到 ④ │ │ │ └──────────────────────────────────────────────────────┘

视觉风格方向

这是 Agent 调试工具，不是 GUI 产品。面向用户的「界面」仍然是终端和代码文件：

  • 运行摘要（终端输出）：灰底白字，绿钩表示测试通过，红叉表示发现问题，黄色警告表示异常行为
  • 结构化错误报告（JSON/YAML 格式）：方便 LLM 解析，也方便用户用 jq 或 yq 过滤
  • 操作回放日志：按时间轴记录「T+0.5s: press KeyD → T+1.2s: collision detected」

整体调性：精密、可信、透明——让用户愿意把「试错」的工作交给 Agent。

6. 风险（非技术）

📉 市场风险

竞品分析：

  • Godot GUT / WAT / gdUnit —— 功能完善的单元测试框架，但需要开发者手动写测试用例，不提供 LLM 集成、日志自动分析和输入模拟
  • Manual QA + Godot Debugger —— 目前的主流方案，完全依赖人。效率和可重复性低
  • Playwright / Puppeteer（Web 游戏） —— 如果游戏导出为 Web 版，可以用 Web 自动化工具测试，但无法与 Godot 项目文件联动分析
  • AI 直接读代码找 Bug（Cursor / Claude） —— 只能做静态分析，看不到运行态。漏报率高

风险： 市场非常小众——「用 AI Agent + 日志 + 输入模拟调试 Godot 游戏」是一个极其细分的需求。同时，如果没有 Godot 官方对 --headless 模式的输入模拟支持，实现成本会很高。

缓解： 从 LLM 分析日志 + 静态代码关联切入（技术门槛低、对用户立即可用），输入模拟作为增强功能随 Godot 官方能力迭代推进。

⚙️ 运营风险

  • Godot 无头模式能力不足 —— Godot --headless 当前可能不支持渲染、无法加载某些资源、或无法响应输入模拟。需深入验证 Godot 4.x CLI 的能力边界
  • LLM 幻觉导致误判 —— Agent 分析日志后可能给出错误的问题定位，进而修改正确的代码，引入新 Bug。需要有「修改前 git commit + 修改后验证失败自动回滚」的安全机制
  • 日志风暴 —— 游戏在高频循环中可能产生海量日志（每帧输出），超出 LLM 上下文窗口。需要智能采样和摘要策略
  • 跨平台兼容 —— Windows/macOS/Linux 下 godot CLI 的输入模拟方式不同（xdotool / SendKeys / CGEvent），需要平台适配

📈 增长风险

冷启动： 调试工具的初始价值取决于「项目质量和调试场景的丰富度」——一个空项目没有日志可分析。

  • 解法： 发布时自带 3 个模板游戏的「预置测试用例」——用户跑一下就知道工具能做什么
  • 解法： 每个模板内置日志注解（log_info / log_error），展示最佳实践的日志埋点方式
  • 解法： 提供一个「一键诊断」命令：对任意 Godot 项目运行、抓日志、出报告。用户不需要提前学任何东西
  • 解法： 在 Godot 社区和 AI 编程社区分享「Bug 诊断 Demo 视频」——直观展示了从报错到修复的全自动过程

💰 变现路径

MVP 阶段：免费 + 开源

  • 作为 Godot Reasonix 技能集的内置模块，随主技能集一起发布
  • 积累用户口碑和使用案例

后续变现（技能集整体商业化时考虑）：

  • LLM API 代理服务 —— 用户不想自己配置 OpenAI / Anthropic API key，可以订阅官方代理服务（按量计费）
  • CI/CD 托管运行器 —— 云端跑测试 + LLM 分析 + 生成报告，月费订阅
  • 企业测试报告仪表盘 —— 聚合多项目测试结果、历史趋势、问题统计（作为 B 端付费功能）

核心逻辑： 调试模块的变现不能依赖「锁定基础功能」——因为调试是开发刚需。变现附加在「规模化」和「AI 算力代理」上。
